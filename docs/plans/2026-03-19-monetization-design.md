# TabVault 商业化设计文档

**Date:** 2026-03-19
**Status:** Approved for planning

## 一、产品定位

### 核心差异化

TabVault 不是"更好的书签管理器"，而是一个本质不同的工具：

> **把书签从"URL存档"升级为"可搜索的知识库"**

**普通书签管理器的假设**：用户记得自己存了什么，只需要"找回"它。
**TabVault 的假设**：用户不记得自己存了什么，需要"理解"它。

传统书签是哑巴档案柜——URL + 标题，语义全无。TabVault 通过 AI 给每条书签注入语义（摘要、标签、可搜索的内容理解），让用户能用自然语言找到三个月前存的任何一篇文章。

### Aha 时刻定义

> "原来我能搜到这个"

用户导入历史书签，AI 批量分析后，用自然语言搜出本以为早已遗忘的内容。这个瞬间是产品的核心价值证明，也是付费转化的触发点。

### 核心优势

1. **语义搜索**：跨 title/summary/tags 全文搜索，找到的是"知识"而不是"URL"
2. **Local-first + BYO API Key**：隐私完全归用户，无数据上传，用户信任基础强
3. **多 AI 提供商**：支持 OpenAI-compatible、Claude、Gemini，用户用已有账户
4. **Chrome 原生书签集成**：增强现有工作流，不是从零开始

## 二、商业模式

### 模式架构

```
┌─────────────────────────────────────────────────────┐
│                  免费试用（3天）                     │
│  内置 key，前50条自动分析，完整体验所有功能           │
└─────────────────────┬───────────────────────────────┘
                      │ 转化
          ┌───────────┴───────────┐
          │                       │
    ┌─────▼──────┐         ┌──────▼──────┐
    │  买断授权   │         │  托管订阅   │
    │  $X 一次性  │    +    │  $Y/月 可选 │
    │  BYO key   │         │  内置 key   │
    └────────────┘         └─────────────┘
```

### 层一：买断授权（核心产品）

- 一次性付费解锁软件永久使用权
- 用户自带 API Key，开发者零运营成本
- 定价参考：$15–$25（对标同类买断工具）
- 支付平台：Paddle 或 LemonSqueezy（支持全球收款、自动处理增值税）
- 激活机制：购买后获得 License Key，在扩展设置页输入激活

### 层二：托管订阅（可选附加服务）

- 不想配置 API Key 的用户按月付费，使用开发者内置的托管 Key
- 本质是 API 转售，定价需覆盖 API 成本 + 利润 + 运营余量
- 定价参考：$5–$8/月
- 与买断独立：买断用户可以选择是否追加订阅；订阅用户也可以随时切换 BYO Key

### 试用期设计

- **时长**：3天（72小时，从首次安装激活计时）
- **额度**：内置 Key 分析前50条书签
- **功能**：试用期间全功能开放，体验与付费版完全一致
- **到期行为**：软件仍可使用（本地已分析的数据不删除），但新的 AI 分析功能需要输入 License Key 或自己的 API Key

## 三、3天试用转化路径设计

### 目标

让用户在72小时内亲眼看到语义搜索找到"本以为找不到"的书签。

### 转化路径

```
安装扩展
    │
    ▼
Day 0: 首次打开 → 一键导入 Chrome 书签 → 自动批量分析（无需配置任何内容）
    │
    ▼
Day 1-2: 用户自然使用，搜索时触发 Aha 时刻
    │         └─ 搜索结果显示"匹配原因"（匹配了 AI 摘要中的 XXX）
    ▼
Day 2: 推送"你的知识库统计"（已分析 N 条，覆盖 X 个主题）
    │
    ▼
Day 3: 试用到期提醒 → 购买引导页（展示已保存的价值，强化付费动机）
```

### 关键设计原则

1. **零摩擦启动**：试用期内置 Key，导入即分析，不问用户任何 API 配置
2. **Aha 时刻可视化**：搜索结果高亮匹配来源（"来自 AI 摘要"/"来自标签"）
3. **价值锚定**：到期前展示数据（分析了多少条、发现了多少主题），让用户感受到"已有的积累"值得保留
4. **非破坏性到期**：已分析的数据全部保留，只是新分析需要付费或自带 Key，降低用户抵触情绪

## 四、实施优先级

按转化路径反推，优先级排列如下：

### P0（试用转化核心路径）

1. **试用期系统**
   - 首次安装时间戳记录（`chrome.storage.local`）
   - 内置 Key 的分析额度计数（前50条）
   - 试用到期状态检测与 UI 提示
   - 试用到期后的优雅降级（不删数据，只限制新分析）

2. **批量分析体验优化**
   - 首次打开引导：一键导入 + 自动触发分析队列
   - 分析进度可视化（已分析 N/总数）
   - 后台静默分析（不打断用户操作）

3. **Aha 时刻增强**
   - 搜索结果显示匹配原因（匹配字段标注）
   - 搜索命中摘要内容时高亮显示片段

### P1（购买流程）

4. **License Key 激活系统**
   - 设置页增加 License Key 输入框
   - Key 验证逻辑（本地验证 or 远端验证）
   - 激活成功后解除分析限制

5. **支付平台集成**
   - LemonSqueezy 或 Paddle 商品配置
   - 购买页面（可以是扩展外的静态页面）
   - Webhook 处理（自动发送 License Key 到买家邮箱）

### P2（订阅附加服务，后期）

6. **托管订阅系统**
   - 订阅状态验证（远端 API）
   - 托管 Key 的用量控制与用户配额管理
   - 订阅管理页面

## 五、技术实现要点

### 试用期系统（本地，无后端）

```ts
// chrome.storage.local 存储结构
type TrialState = {
  installedAt: string        // ISO 时间戳，首次安装时间
  analysisUsed: number       // 试用期已用分析次数
  licenseKey?: string        // 激活的 License Key
  licenseValidated?: boolean // Key 是否已验证
}

// 试用状态判断
function getTrialStatus(state: TrialState): 'trial' | 'expired' | 'licensed' {
  if (state.licenseKey && state.licenseValidated) return 'licensed'
  const elapsed = Date.now() - new Date(state.installedAt).getTime()
  const threeDays = 3 * 24 * 60 * 60 * 1000
  if (elapsed > threeDays || state.analysisUsed >= 50) return 'expired'
  return 'trial'
}
```

### License Key 验证方案

**方案 A（无后端）**：离线验证
- 使用 LemonSqueezy 的 License Key 格式，本地用公钥验证签名
- 优点：无需维护后端；缺点：无法吊销 Key

**方案 B（轻量后端）**：远端验证
- 激活时向 LemonSqueezy API 验证 Key，结果缓存本地
- 每隔30天重新验证一次
- 优点：可吊销、可统计激活数量；缺点：需要网络

**推荐方案 B**，LemonSqueezy 提供开箱即用的 License Key API，实现成本低。

## 六、成功指标

- **试用安装 → 付费转化率**：目标 >5%（买断软件行业参考值）
